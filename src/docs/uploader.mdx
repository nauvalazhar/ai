A controller that pairs with `<Attachment>` to handle the upload side: picking files, validating, dragging, dropping, pasting, tracking progress, and surviving a reload. Renders no chrome of its own. You bring the trigger, the dropzone, the list rows, and the avatar tile; the kit owns the queue, the input, the progress plumbing, and the abort wiring.

## Anatomy

The root sits high in the tree. It owns the file input element, the queue state, and the abort controllers. Triggers, dropzones, lists, and previews live anywhere underneath and read the queue through `useUploader()`.

```tsx
<Uploader uploader={upload}>
  <UploaderTrigger render={<Button><PaperclipIcon /> Attach</Button>} />
  <UploaderList>
    {(item, actions) => (
      <Attachment progress={item.status === "uploading" ? item.progress : undefined}>
        ...
      </Attachment>
    )}
  </UploaderList>
</Uploader>
```

The dropzone is optional. Drop it in for a drag target, leave it out for a button-only flow.

```tsx
<Uploader uploader={upload}>
  <UploaderDropzone className="border-2 border-dashed p-8 rounded-outer text-center">
    Drop files here, or
    <UploaderTrigger render={<Button variant="ghost">browse</Button>} />
  </UploaderDropzone>
  <UploaderList>{(item, actions) => <Attachment ...>...</Attachment>}</UploaderList>
</Uploader>
```

## The uploader function

You write it. The kit calls it once per item.

```ts
type UploaderFn = (input: {
  id: string;
  file: File;
  signal: AbortSignal;
  onProgress: (progress: number) => void;
}) => Promise<unknown>;
```

The id lets you key external state by item, which matters for retries and resumable flows. The signal aborts when the user cancels or removes the item; wire it into your transport. The progress callback takes a number in `[0, 100]`. The resolved value lands on `item.result`. If it has a string `url` field, the kit copies it into `item.previewUrl` so reloaded items can show a thumbnail without holding the original `File`.

For the common case of one HTTP request per file with multipart body and upload progress, use the bundled helper.

```ts
import { createXhrUploader } from "#/components/ai/uploader";

const upload = createXhrUploader({
  url: "/api/upload",
  field: "file",
  headers: () => ({ Authorization: `Bearer ${getToken()}` }),
});
```

For anything custom (signed URLs, chunked transfer, your own protocol), write the function yourself.

## Hooks

### useUploader

Read the queue and act on it from any descendant. Returns the same shape regardless of controlled or uncontrolled mode.

```ts
const { items, open, add, cancel, retry, remove, clear } = useUploader();
```

`open` opens the hidden file picker. `add` accepts a `File[]` or `FileList` and pushes through the same validation the trigger uses. The action methods take an item id and do the right thing for that item's current status. The items array reflects live progress merged in, so a render-time read of `item.progress` is always fresh.

### useUploaderAttach

Attach drop and paste behavior to an arbitrary element. Useful when an existing component (a composer, a prompt, a canvas) should accept files as a side behavior without becoming a dropzone visually.

```ts
const ref = useRef<HTMLDivElement | null>(null);
const { isDragOver } = useUploaderAttach(ref, { drop: true, paste: true });
```

The hook installs listeners on the element behind the ref, ignores drags that do not carry files, and counts drag depth so nested children with their own drag handlers do not flicker the over state. The returned `isDragOver` is the same signal you get on a `<UploaderDropzone>` and is suitable for an overlay.

## Item shape

```ts
type UploadItem = {
  id: string;
  name: string;
  size?: number;
  type?: string;
  status: "pending" | "uploading" | "success" | "error" | "canceled";
  progress: number;
  file?: File;
  previewUrl?: string;
  result?: unknown;
  error?: { code: string; message: string };
};
```

The `file` field is the volatile half. It exists in the session the file was picked in and is gone after a reload. Everything else is durable. To restore a previous queue, hydrate items with the durable fields, `status: "success"`, and `progress: 100`. The kit does not auto-start items that have no `file`.

## API

Reference for each part of the component, including its available props and behavior.

### Uploader

The root. Owns the queue, the abort controllers, and the hidden file input. Renders no DOM of its own beyond the input, so layout decisions stay with the consumer.

The kit treats `onItemsChange` as the structural channel and `onProgress` as the granular one. Wiring `onItemsChange` to a storage write is safe; it only fires when an item is added, removed, retried, cleared, or transitions status. Progress ticks come through `onProgress` and the merged `items` array on the context.

Controlled mode follows the standard React shape: pass `items` and `onItemsChange`, then forward the new value back into state from the handler. Uncontrolled mode uses `defaultItems` for the initial snapshot and emits the same `onItemsChange` for storage.

### UploaderTrigger

A passthrough button that opens the file picker. Pass the visible element through `render`. The kit attaches an `onClick` and merges with any handler on the rendered element, so a Button or a MenuItem keeps its own behavior.

## Persistence

Reloaded queues are user-implementable today; the kit ships no storage adapter. Two rules cover the common cases.

First, only persist `success` items. Anything in flight at the time of reload has no `File` to resume against and is not meaningful to restore. Second, strip the `file` field before serialization, since `File` does not survive `JSON.stringify`.

```tsx
const [initial] = useState(() => {
  const raw = localStorage.getItem("files");
  return raw ? JSON.parse(raw) : [];
});

<Uploader
  uploader={upload}
  defaultItems={initial}
  onItemsChange={(next) => {
    const persistable = next
      .filter((i) => i.status === "success")
      .map(({ file, ...rest }) => rest);
    localStorage.setItem("files", JSON.stringify(persistable));
  }}
/>
```

For an avatar that survives reload, the same pattern with `maxFiles={1}` and a single-item store works.

## Resumable uploads

The kit does not ship a resumable transport, but the surface gives you the seams to build one: a custom `uploader` function that owns the protocol, controlled `items` with `onItemsChange` for persistence, and `retry(id)` to restart. The `id` passed into your `uploader` is the key for an external session store.

The shape below uses a tus-style request flow. Swap in S3 multipart or your own scheme without touching the kit.

```tsx
type Session = {
  fingerprint: { name: string; size: number; lastModified: number };
  uploadUrl: string;
  bytesUploaded: number;
};

const sessions = new Map<string, Session>(loadSessions());

const upload: UploaderFn = async ({ id, file, signal, onProgress }) => {
  const prior = sessions.get(id);
  const sameFile = prior && matches(prior.fingerprint, file);

  let uploadUrl: string;
  let startByte = 0;

  if (prior && sameFile) {
    uploadUrl = prior.uploadUrl;
    const head = await fetch(uploadUrl, { method: "HEAD", signal });
    startByte = Number(head.headers.get("Upload-Offset") ?? 0);
    onProgress((startByte / file.size) * 100);
  } else {
    const res = await fetch("/api/upload", { method: "POST", signal });
    uploadUrl = res.headers.get("Location")!;
    sessions.set(id, { fingerprint: fingerprint(file), uploadUrl, bytesUploaded: 0 });
    persistSessions();
  }

  const CHUNK = 5 * 1024 * 1024;
  for (let offset = startByte; offset < file.size; offset += CHUNK) {
    if (signal.aborted) throw new Error("aborted");
    const chunk = file.slice(offset, Math.min(offset + CHUNK, file.size));
    await fetch(uploadUrl, {
      method: "PATCH",
      headers: { "Upload-Offset": String(offset) },
      body: chunk,
      signal,
    });
    sessions.get(id)!.bytesUploaded = offset + chunk.size;
    persistSessions();
    onProgress(((offset + chunk.size) / file.size) * 100);
  }

  sessions.delete(id);
  persistSessions();
  return { url: deriveFinalUrl(uploadUrl) };
};
```

On rehydration, mark any item that has a session but is not in `"success"` as `"error"` with a code that distinguishes it from a real failure. Show a Resume button on those rows. When the user picks the file again, splice it into the existing item and call `retry`, so the item id (and the session keyed off it) survives.

```tsx
function resume(id: string) {
  const input = document.createElement("input");
  input.type = "file";
  input.onchange = () => {
    const file = input.files?.[0];
    const session = sessions.get(id);
    if (!file || !session) return;
    if (!matches(session.fingerprint, file)) return;
    setItems((prev) =>
      prev.map((i) => (i.id === id ? { ...i, file, status: "pending" } : i)),
    );
    queueMicrotask(() => retry(id));
  };
  input.click();
}
```

Things to keep in mind: a wrong file on resume should be refused or treated as a fresh upload, sessions can outlive items so the server-side store needs an expiry, and cancel mid-upload should keep the session row around so the user can resume later instead of restarting from zero.

Future kit additions (`meta` on `UploadItem`, `previous` on the uploader input, `onMeta` for mid-flight writes, `attachFile(id, file)` as a one-call swap-and-retry) will shorten this. Code written against today's surface keeps working when they land.
